# 白泽 (Baize) 迭代核心原则

> 本文档定义系统的设计哲学与迭代方向。技术护栏（数据溯源/时效性/交叉验证等）见 `.claude/rules/guardrails.md`。
> 每次重大迭代决策前，回看本文档确认方向一致性。

---

## 原则一：帮助投资者建立自己的交易系统（最高优先级）

**系统的终极目标不是让用户依赖系统，而是让用户不再需要系统。**

### 为什么

- 真正好的投资教练，教的是核心原则，不是一套永远无法内化的复杂系统
- 如果用户用了 2 年还是每天打开系统问"能不能买"，系统作为教练是失败的
- 系统输出必须适配用户的风格、偏好、能力圈——不是系统告诉用户该做什么，而是系统帮助用户做出更好的、符合自身风格的决策

### 如何做到

1. **每一层输出都带精简版**：「如果你要自己判断，只看这 3 个指标」——完整输出 20 个指标 + AI 报告，精简版是「PE 分位 + ROE 趋势 + 北向资金方向」
2. **分层启用**：小白模式（15 条军规全部强制）→ 进阶模式（block 级保留，warn 级可配置）→ 专业模式（仅 4 条核心 block 规则）。随着能力提升逐步解锁自由度
3. **独立决策追踪**：Phase 4 增加「独立决策准确率」指标——用户未受系统建议影响的独立决策，事后胜率如何。连续 6 个月 > 55% → 祝贺用户已可独立投资
4. **退出策略内置**：系统在设计之初就设定自己的退出路径——从「决策助手」→「风控安全带」→「可选参考」
5. **不替用户做决策**：系统输出分析和信号，但最终决策权在用户。系统记录用户的每次决策与事后结果，帮助用户建立自己的决策框架

### 如何衡量

- 用户的独立决策准确率是否在提升
- 用户对系统建议的依赖度是否在下降（不是不用，而是有自己的判断）
- 用户是否能说出自己的交易系统的核心规则

## 原则二：阻止自毁优先于捕捉机会

**对于多数投资者，阻止一次冲动交易的价值 > 推荐 10 只好股票的价值。**

### 为什么

- 小白看到 ⚠️「不建议追高」，然后看着股价继续涨，心里想「这破系统不行」，手动追进去——系统的输出如果用户不执行，等于不存在
- 真正能保护用户的不是推荐，是拦住不该做的操作

### 如何做到

1. 军规 block 级规则必须在操作路径上**物理阻断**——不是「建议不要买」，而是「系统拒绝执行此操作」
2. 配合事后对比——用户手动操作被事后打脸——逐步建立信任
3. 系统的默认立场是保守，用户随着能力提升逐步解锁

## 原则三：数据质量是基石，垃圾进垃圾出

**如果华泰说 PE=15.2，国信说 PE=16.8，AKShare 说 PE=14.5——L1 分析吃的是互相矛盾的数据，L2 法官的评分就是垃圾。**

### 为什么

- 量化分析的质量上限由数据质量决定
- A 股数据源差异比美股大得多

### 如何做到

1. 同一数据点 ≥ 2 源交叉验证，差异 > 5% 标记 DISPUTED，不进入分析管道
2. 数据溯源三要素必标注：tier（T0-T3）+ nature（fact/interpretation/speculation）+ freshness
3. 缺失/失败数据源标记 `[DATA_GAP]`，下调对应维度权重
4. Phase 1 必须完成 A 股全市场数据一致性审计

## 原则四：敬畏市场

**市场永远是对的。任何模型、策略、分析框架都是对市场的简化近似，而不是市场本身。**

### 为什么

- 回测好 ≠ 未来好。过去有效的因子可能已经失效，过去没见过的尾部风险随时可能发生
- 连续盈利是最危险的状态——它让人产生"我懂了"的幻觉，而市场正在酝酿下一次颠覆认知的波动
- A 股的制度特征（政策驱动、散户主导、涨跌停限制）意味着很多尾部风险比美股更大、更频繁
- 模型的置信度区间是对过去的拟合，不是对未来的承诺

### 如何做到

1. **永远预留安全边际**：无论模型多看好，仓位上限不可逾越；无论回测多漂亮，永远假设最差情况可能更差
2. **胜率越高越警惕**：连续盈利触发风控收紧——不是惩罚成功，而是承认"好运气不会一直持续"
3. **模型自我质疑机制**：L2 法官裁决阶段必须输出「如果这个判断是错的，最可能的原因是什么」——强迫系统从反面审视自己的结论
4. **定期压力测试**：用历史极端行情（2015 股灾、2016 熔断、2020 疫情跌停潮、2024 年初流动性危机）对当前策略做回溯，检验在最坏情况下是否扛得住
5. **不为短期正确膨胀**：单次预测准确不改变系统对自身能力的评估。置信度校准必须基于长期统计，不能因为"最近几次都对"而上调

### 如何衡量

- 策略在极端行情压力测试中的最大回撤是否可控
- L2 裁决的置信度校准是否准确（预测 80% 信心的事件，实际发生频率是否接近 80%）
- 连续盈利后系统是否自动收紧了风控参数
- 是否定期输出了「自我质疑」报告

## 迭代决策检查清单

每次重大修改/新增功能前，逐条确认：

- [ ] 是否让用户更依赖系统而非更独立？（违反原则一）
- [ ] 是否优先保护用户而非优先捕捉机会？（违反原则二）
- [ ] 新增数据源是否经过交叉验证？（违反原则三）
- [ ] 是否因为近期表现好而放松了安全边际或风控约束？（违反原则四）
